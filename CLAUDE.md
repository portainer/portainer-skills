# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

Agent skills for integrating with Portainer. Each top-level directory is one self-contained skill, installed by copying it into a skills location (e.g. `~/.claude/skills/<name>`). Releases are tagged to match the Portainer version whose OpenAPI spec they bundle (e.g. `2.42.0`).

There is currently one skill, `portainer-api`, which teaches an agent to drive the Portainer REST API from curl or any plain-HTTP client.

## Layout (`portainer-api/`)

| Path | What it is |
|---|---|
| `SKILL.md` | Entry point — auth, URL anatomy, response discipline, error decoding, spec navigation |
| `references/*.md` | Domain guides: environments, Docker proxy, Kubernetes proxy, stacks |
| `references/spec/` | The Portainer OpenAPI spec, chunked per API tag (committed; regenerated by `make specs`) |
| `references/spec/INDEX.md` | Generated operation index — one greppable line per operation |
| `scripts/`, `Makefile` | Spec refresh pipeline |

## Commands

There is no build or test suite — the repo is markdown plus a spec refresh pipeline.

Refresh the bundled spec (run from `portainer-api/`; requires SSH access to `portainer/portainer-api-docs` and `uv`):

```bash
make specs VERSION=2.42.0
```

This sparse-clones the version's per-tag chunk files into `upstream/` (gitignored), copies them into `references/spec/`, then post-processes them. The post-processing step can be re-run alone:

```bash
uv run --with pyyaml scripts/postprocess_specs.py references/spec --version 2.42.0
```

It strips the `info.description` preamble repeated in every chunk, filters each chunk's `tags:` list to the tags it actually uses, drops the Redoc-only `x-tagGroups:` block, regenerates `INDEX.md`, and re-parses every chunk — a parse failure aborts non-zero so a broken fetch can't be committed silently.

## How the skill is structured

The skill is built on progressive disclosure — an agent loads only what the task needs:

1. `SKILL.md` frontmatter description — triggering.
2. `SKILL.md` body — core rules plus a routing table mapping each API area to its spec chunk.
3. `references/*.md` — per-domain guides, read only when the task touches that domain.
4. `references/spec/*.yaml` — one self-contained chunk per API tag (`$ref`s never leave the file), resolved via a grep on `references/spec/INDEX.md` rather than read whole.

## Editing rules

- `references/spec/` (including `INDEX.md`) is generated — never hand-edit it. Fix the upstream spec or `scripts/postprocess_specs.py` and re-run `make specs`.
- Hand-written guidance lives in `SKILL.md` and `references/*.md`. Follow their style: explain *why* a rule exists (the failure mode it prevents), not just the rule.
- Edition terminology: use **EE (Enterprise Edition)** and CE — never "BE"/"Business Edition".
- The bundled spec is EE, the superset; `SKILL.md` documents how EE-only paths 404 on CE. Keep that framing when touching edition-related guidance.
